架构设计

当前版本：自包含 HTML + JSON 数据源
数据层

JSON 数据源

管理页面使用独立的 JSON 文件（site/data/worklog.json 等）作为数据源，通过 Jinja2 或浏览器 JS 渲染为 HTML。无数据库、无构建步骤。
渲染层

模板化生成

落地页通过 templates/landing/ 下的 Jinja2 模板生成；管理页通过 CLI 更新 JSON 后重新渲染。
部署层

GitHub Pages 静态托管

所有产物均为静态 HTML/CSS/JS，通过 GitHub Actions 自动部署到 gh-pages 分支。无服务器维护。
交互层

Claude Code Skill + CLI

Skill 负责解析用户意图并调用底层 CLI；CLI 负责实际文件操作。用户也可脱离 Skill 直接使用 CLI。

页面类型一览

四套模板对应不同增长场景
类型 A — 核心转化页

Type A

主题：暗色优先 + 高对比 accent
结构：Hero + 价值主张 + 功能网格 + FAQ + CTA
用途：高搜索量核心关键词（如 /ai-music-video-generator）
类型 B — 单工具页

Type B

主题：同暗色优先
结构：工具入口 + 交互演示 + 使用步骤 + 相关工具
用途：具体工具关键词（如 /tools/band-name-generator）
类型 C — 工具入口/Hub 页

Type C

主题：同暗色优先
结构：指标卡 + entry-card 网格 + 子页面导航
用途：聚合入口（如 /tools/audio-to-video）
Article — 文章页

Article

主题：同暗色优先，阅读友好
结构：面包屑 + 标题 + 元信息 + 正文 + 相关推荐
用途：教程/榜单/对比类内容

AI 更新操作规程

1

定位目标页面

确定要修改的管理页，打开对应的 site/*.html 文件或其数据源 site/data/*.json。

2

更新数据源

在 JSON 文件中增加、修改或删除记录。优先通过 CLI 命令 ogi site update --page worklog --entry "..." 完成。

3

重新渲染 HTML

CLI 在更新 JSON 后自动调用 Jinja2 重新生成对应 HTML。本地用浏览器打开验证。

4

更新统计数字

如修改影响首页统计，同步更新 site/index.html 中的 .stat-pill .num。

5

运行敏感信息扫描

提交前执行验证清单中的 grep 命令，确认无密钥、无本地路径、无真实品牌名残留。

工作记录新增规范

worklog.html records[] 数组，插入到最前
①

字段结构（每条记录必填）

id：全局唯一，格式 WL-[主题关键词][MMDD]
time：YYYY-MM-DD
action：create | update | delete | confirm
entity：业务实体分类（如 growth-site / landing-builder / competitor-report）
name：动作-对象-版本/范围（简洁，15字内）

②

内容字段

description：一句话摘要
purpose：为什么做
include：本次实际包含的范围
exclude：明确不包含的边界
conclusion：核心判断或结果
todo：下一步待办（无则留空）
hack：临时处理、已知遗漏或风险（无则留空）

③

impact 影响文件数组

impact: [ { path: "相对项目根的路径", note: "新建/更新/说明" } ]

④

search 搜索补全

补充该条记录的关键词，空格分隔，供搜索框命中。包含主题词 + 日期 + 涉及文件名关键词。

全局导航维护规范

全站统一 3 项

3 项固定导航（所有页面 topbar 右侧）

总览（index.html）· 工作记录（worklog.html）· 维护说明（maintenance.html）
新建任何页面时必须在 topbar 加入此 nav，路径根据当前文件深度调整 ../ 层级。

HTML 属性引号规范

class / style / href / src / id / target / rel 等属性值边界必须用英文半角 "..."，严禁使用中文弯引号。
每次改 HTML 后用 rg -n 'class=[""]|style=[""]|href=[""]' 文件路径 检查，无结果才算合格。
